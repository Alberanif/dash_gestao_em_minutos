Como reestruturar o projeto para múltiplas contas?

O banco atual não tem suporte a múltiplas contas. Precisamos decidir como evoluir o schema e a arquitetura.

NOVA TABELA: accounts
id · platform · name · credentials · is_active
MIGRAÇÃO: adicionar account_id
youtube_channel → + account_id FK
youtube_videos → + account_id FK
instagram_profile → + account_id FK
instagram_media → + account_id FK
Dados existentes migrados para a conta inicial

A — Migração incremental ✨ Recomendado

Adiciona uma tabela accounts e uma coluna account_id nas tabelas existentes. Preserva dados já coletados e evolui o schema sem quebrar nada.

Prós

  • Preserva dados existentes
  • Menor risco
  • Evolução natural

Contras

  • Migration script mais trabalhoso
DROP das tabelas atuais
↓ remove tabelas sem account_id
CRIA schema novo limpo
accounts (platform, credentials...)
youtube_channel_snapshots (account_id, ...)
youtube_video_snapshots (account_id, ...)
instagram_profile_snapshots (account_id, ...)
instagram_media_snapshots (account_id, ...)

B — Schema limpo do zero

Descarta as tabelas atuais e recria tudo com account_id desde o início. Schema mais limpo e consistente. Ideal se não há dados reais ainda.

Prós

  • Schema perfeito desde o início
  • Sem legado

Contras

  • Perde dados existentes
  • Reescreve mais código
youtube_igt_channel_snapshots
youtube_igt_kids_channel_snapshots
instagram_igt_profile_snapshots
instagram_igt_kids_profile_snapshots
Uma tabela por conta, prefixada com o nome

C — Tabelas separadas por conta

Cada conta gera seu próprio conjunto de tabelas no banco. Sem foreign keys, sem migrations complexas. Mas escala muito mal e é difícil de manter.

Prós

  • Zero conflito entre contas

Contras

  • Não escala
  • Código duplicado
  • Difícil de manter
